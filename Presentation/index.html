About Angular

  • MV* (Model View Whatever)
    • A nice way to organize code by single responsibility, and decouple DOM, logic and data.
  • Expands HTML
    • Allows us to be declarative, and avoid a lot of procedural javascript. DOM manipulation is segregated. <evo1-textbox placeholder="enter name here"/>
    <div evo1-mouseovereffect></div>
  • Provides structure for properly factored code
    • Services, controllers, directives, filters, factories, etc. allow for beautiful composition.
  • From Google
    • Created in 2009 at a startup, open-sourced, and adopted by Google where startup partner was employed.

How Angular Works

  • DI-Driven and Testable
    • Angular encourages separation into different composable parts, which can be injected as dependencies into each other. These parts can be easily tested or mocked.
  • Scope
    • Scope wraps a hierarchical layer of the DOM, and is inherited by child scopes. This is a reflection of the prototypal inheritance of javascript. There will probably be many different scopes in play in an application.
  • Dirty-checking vs. change listening
    • Scope variables are registered to be watched. The Angular "digest cycle" runs through all these variables looking to see what changed. By default, data-bound variables are watched. Change listening is a common approach in other libraries, observable objects register listeners.
  • Can use POJOs vs. "observables"
    • For other libraries, you must inherit from an observable to be able to bind data. Using POJOs means less code which is less complex.
  • Framework vs. Library
    • Many popular libraries are not really frameworks in the true sense. Knockout does databinding and jQuery does DOM manipulation, but neither offer a philosophy for building an application. Angular encourages a specific architectural approach and style.

A Different Paradigm

  • SPA
    • Angular apps are often a "single page application", with HTML fragments and server-side data retrieved and displayed just-in-time. They are more like a desktop application, and more responsive. Routing allows deep-linking and the ability to return to a specific state, based on URI.
  • No (or very little) server-generated code
    • The browser requests static html, css and js. All display logic is in js, and all data is delivered as needed via AJAX calls. Very efficient delivery - can even use CDN for everything but API. This can also increase security by not having code executed on the server.
  • AJAX/REST
    • The usual approach is to consume one or more REST APIs for data. REST is a lightweight and efficient web service methodology. Angular can also consume other web service types, such as SOAP.

Setting Up Your Environment

  • Writing Code
    • You can use your favorite editor or IDE to build an Angular application - WebStorm, Sublime, VS, etc., and use supporting tools to serve, debug, run and test.
  • NodeJS/NPM
    • NodeJS is a javascript runtime environment that can act as a back-end for applications. It can run a web server to use in development. Node Package Manager, which is like NuGet for NodeJS, can download libraries such as Express and Bower. NPM can also run tasks like starting your dev server and running tests.
  • Bower
    • Bower is like NPM for the client, managing dependencies for your web application, such as Angular, jQuery, Underscore, etc.
  • Test runners
    • Karma is the official Angular js test runner. Chutzpah (available via NuGet) integrates very well with Visual Studio.
  • Plugins
    • Many plugins for code completion, snippets, etc. exist for Resharper and different editors.
  • NuGet
    • Can add Angular core, NodeJS, more from NuGet.
  • NTVS
    • Node Tools for Visual Studio provides NodeJS conveniences for Visual Studio users, such as project templates and the ability to quickly F5/debug your NodeJS server.

Angular Application Structure

  • Modules
    • A module is a collection of related code that can be added as a dependency to another module. A module is not directly related to a file, and are often comprised of multiple files. Modules are kind of like assemblies.
  • Dependency Injection
    • Within modules, there are controllers, services, directives, filters and other "entities" that perform a specific duty. All of these entities are registered with the Angular application, and may declare dependencies on each other. When a dependency is declared, Angular will inject that dependency at runtime.
  • Angular Application
    • The Angular application is the highest level module, and composes constituent modules. There may be more than one in your web application, though it is not common.

Controllers

  • Controllers Connect Things
    • The controller is the "glue" that connects the various concerns. Common controller tasks are initializing state, retrieving data/binding to the model, and calling out to Angular services in our app. The controller should do no DOM manipulation and contain no logic. As a result, it should remain small.
  • Scope
    • Controllers can cover an entire application or one div, depending upon where it is declared. Controllers connect functions and data to the scope, to be used from the UI.
  • Hierarchical
    • Controllers can be nested. For instance, you can have a SecurityController that manages authentication for your entire application, and more conventional controllers within it.
  • Initialization
    • Controllers can be declared in markup as an element attribute, or in route configuration.

  • Enable separation of concerns

Filters

  • Two main types - formatting and collection filtering
  • A bunch of built-ins for dates, currency, etc., and you can make your own
  • Piping through filters for formatting
    • {{claims.amount | currency}}
  • Piping through filters for formatting - great for images too
    • {{claim.status | statusicon}}
  • Piping through filters for collection filtering
    • {{claims | filter:criteria}}
  • Can sort
    • {{claims | orderBy:'category'}}
  • Can chain filters
    • {{claims | orderBy:'category' | filter:criteria}}
